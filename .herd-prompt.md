# Mission

You are an autonomous engineer. Resolve the GitHub issues in the GROUP section
(appended below this file) and deliver ONE pull request a reviewing agent can assess
standalone. Your working copy is a dedicated git worktree already checked out on your
feature branch. Never switch to, commit on, or push `origin/main`'s branch.

## Non-negotiable process

1. For EACH issue run `gh issue view <N> --comments` FIRST. Where a triage/burn-down
   comment disagrees with the issue body, the triage comment wins.
2. Re-verify every claim against THIS checkout before coding — cited lines may be stale.
3. If an issue is already fixed or refuted on the current branch: do NOT write code —
   comment with concrete evidence (file:line, test output) and `gh issue close <N>`.
4. Fix root causes. No stubs, no NOPs, no hacking around a failing test. Every fix gets
   a regression test that fails before your change and passes after; say so in the PR.
   Put new tests in a NEW test module named for your work (e.g. `tests/test_<topic>.py`)
   instead of appending to a shared test file — other workers are editing those files in
   parallel and appended tests collide at merge time. Only touch a shared test file when
   the repo's test conventions leave no alternative.
5. Liveness discipline: bn-lens = a headless Binary Ninja navigator TUI (Rust + ratatui/crossterm), packaged as a
herdr plugin. It shells out to `bn` (the Binary Ninja bridge CLI from /opt/bn) and `herdr`;
there is NO direct BN API usage in this repo. src/bn.rs and src/herdr.rs are the ONLY
modules that shell out — keep it that way.

TESTS: only pure helpers are unit-tested (inline #[cfg(test)] modules — e.g. syntax.rs
tokenizer, viewer/hotspots.rs). The TUI itself is dogfooded in a herdr pane, not
integration-tested. New pure logic you add or change needs inline unit tests in the module
you touched; UI/threading behavior that cannot be unit-tested must be justified in the PR
(what you verified by build + reasoning). Do NOT invent an integration harness.

INVARIANTS (from CLAUDE.md — violating these is an automatic reject):
- Narrow write surface: rename / comment (`;`) / bookmark-tag (`t`) / type declaration only,
  all via `bn`, all live-in-instance. NEVER call `bn save`.
- Agent asks are fail-closed (only the launching pane, verified session) and single-line.
- Render through ratatui's Buffer (clips instead of panicking) — never bypass it.

SANITIZE TEST DATA — HARD RULE: this tool is dogfooded against real proprietary/firmware
targets. Never put real target names, instance IDs, real symbol names, concrete addresses,
or verbatim decompile from a target into commits, PR bodies, issue comments, or test
fixtures. Invent plausible, internally consistent mock data instead.

Upstream `bn` CLI changes are OUT OF SCOPE for this repo (/opt/bn is a separate repo).
If a root cause lives upstream, fix defensively lens-side and note the upstream issue in
the PR body.
 Confirm a code path is reachable before
   claiming a "live" fix; label dead-code fixes LATENT in the PR.
6. Keep the diff scoped to your issues. Do not touch tracking/state files or unrelated code.

## Reporting back — how the orchestrator reads your status

The orchestrator never reads your messages. It polls GitHub and your pane's agent
state, so you report by changing world state:

- SUCCESS = an open PR whose head branch is exactly this worktree's feature branch.
  Never create or switch to another branch — a PR from any other branch is invisible
  to the orchestrator and you will be recorded as failed.
- ALL-REFUTED = no PR and EVERY issue in your group closed (`gh issue close` with
  evidence). If even one issue stays open and there is no PR, you are recorded as failed.
- Stopping to ask a question parks you as BLOCKED, waiting indefinitely for a human.
  Only stop when you genuinely cannot proceed; otherwise make the call and document
  it in the PR body.
- Your tab is closed shortly after your PR is detected. Finish ALL issue comments and
  closes BEFORE `gh pr create`; make the PR and your final summary your last actions.

## Issue #5 — Function-doc comments (bare `;` on a function) do not appear in Marks
Verify then fix-or-close.

- Root cause is documented in `src/marks.rs:7-10` and TODO.md: a bare `;` on a function
  writes BN's function-documentation comment (`fn.comment`), but `bn comment list`
  enumerates only address-scoped comments, so Marks never shows it.
- Confirmed: the installed `bn comment list` has NO `--include-docs` (or similar) flag —
  option 3 from the issue (extend `bn comment list`) is out of scope (/opt/bn is a
  separate repo). Choose between:
  1. **Synthetic entry-address comment**: after setting a function-doc via `;`, also set
     an address-scoped comment at the function entry so it shows in Marks (and in
     `bn comment list` for every client). Tradeoffs: double-render risk in the decompile
     (the `//` doc line plus an entry-line comment), and it's a write-path change — must
     stay within the narrow write surface (single `bn comment set`, live-in-instance,
     NEVER `bn save`).
  2. **Session-local merge**: keep a lens-side record of function-docs set this session
     and merge them into the Marks build. Read-only for BN state, but docs set in
     previous sessions or by other clients stay invisible, and the record dies with the
     process.
  You own this decision — weigh the tradeoffs against the issue's "Expected" section,
  document the choice and rejected alternative in the PR body. (A hybrid is also
  legitimate: session-local merge now, labeled as partial with `Addresses #5`, plus a
  clear-comms fallback — e.g. the `;` flow telling the user the doc won't appear in
  Marks.)
- Whatever you pick, update the stale module doc in `src/marks.rs:7-10` and the TODO.md
  entry to match the new behavior.
